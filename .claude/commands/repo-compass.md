# repo-compass — 「次に何を新しく作るか」を repo の自画像から導く

## これは何で、何でないか

- **これ**: `~/Developer/llm/` 全体を user が作ってきたもの = revealed preference (言語化より忠実な世界観の現れ) として読み、**欲望の軌跡が次に示唆する新規ビルド**を出す。
- **これでない**: feature-loop (= per-project の「破壊せず便利にする」改善案)。repo-compass は per-project の改善でなく、**横断で「次に何を新しく作るか」**。両者は別レイヤ。

## 絶対に避ける崩落モード (このセッションで実際に全滅した出し方)

1. **asset-wiring**: 「既存 A × 既存 B を配線して %削減」。→ 禁止。価値を工数% で語らない。
2. **20個サーベイ**: 尤もらしい候補を大量に並べて user に選ばせる。確信欠如のサインで [[feedback_no_user_pick_from_self_options]] 違反。→ 少数 (3本) に絞り、推しを明示。
3. **部品箱読み**: projects/ を「配線できる部品の在庫」として読む。→ 自画像として読む。同じファイルを正反対のレンズで。
4. **外部 friction の推測**: 「毎月手で再計算してるはず」等、観測してない苦労をでっち上げる。→ 軌跡 (実際に作り・更新したもの) に接地。

## 手順

### 1. revealed preference 信号を集める (deterministic・cheap)

user 定義の対応 (memory SSoT) を proxy で測る:
- **興味の度合い** ← 内容の充実度 + 更新頻度。`~/.claude/runtime/stall_ledger.json` (stall_detector が per-unit の最終編集を保持) + 直近 30/90 日の Edit/Write 頻度。
- **キャパ / 趣味の数** ← 同時編集している unit 数 (直近窓で動いている unit のクラスタ)。
- **メタ理解度** ← 内容の精緻度 + 調査の深度 (reference/ の量, research notes 数, doc 行数, datamine の有無)。
- **新しさ / 軌跡** ← unit の誕生時期と熱の推移 (最近生まれて加速 = 新しい attractor / 昔から高頻度 = 中核 / 冷えつつある = 関心の移動)。

粒度は CLAUDE.md を持つ最近接 unit (project でなく sub-project; stall_detector と同じ単位)。

### 2. 自画像として読む — 組織原理を1つ名指す

信号を眺めて「この repo 全体が何を欲しているか」の通奏低音を1文で言う。
(例: このセッションでの発見 =「暗黙を可視にする = 自分のやっていること(遊び/仕事/注意/学習)を見て測って改善できる形に変える自己観測の器を作り続けている」)。
個別の機能でなく、**横断する欲望の方向**を当てる。

### 3. 軌跡を外挿 → 3本の確信ある賭け

組織原理の「次に示唆するもの」を **3本だけ** 出す。各々:
- **場面**: user が「それ欲しい」と認識する具体シーン (抽象でなく、user の生活/作業の1場面)。
- **なぜ軌跡から**: どの信号 (どの unit の熱/深さ/誕生) がこれを示唆するか。
- **確度**: 「資産・挙動の実在 (高めが多い)」と「接続で価値が出る仮説 (中が多い)」を分けて書く。

### 4. 推しを1つ + 自己批判

3本のうち推しを1つ、逆算した便益込みで明示。さらに最低1本は自分で弱点を暴く (このセッションで賭け1 を「単体では vanity」と下方修正したように)。サーベイにしない。

## 出力

- 通奏低音 1文 → 3本の賭け (場面/軌跡/確度) → 推し1つ。日本語。
- user が1本を指したら、そこから先 (設計 → 実装) は通常フローへ。この command は **生成まで**。
